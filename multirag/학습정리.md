## 1. 배경 지식 — RAG란 무엇인가?

### AI가 질문에 답하는 두 가지 방법

**방법 A: 기억만으로 답하기 (순수 LLM)**

ChatGPT 같은 AI는 수천억 개의 문장을 학습해서 지식을 "뇌 속에" 압축해 저장합니다.  
하지만 이 방식에는 치명적인 한계가 있습니다.

- 학습 이후 생긴 새로운 정보는 모른다 (지식 단절)
- 정확한 숫자나 출처를 물어보면 그럴듯하게 꾸며낸다 (할루시네이션)
- 내 회사 내부 문서, 비공개 데이터는 절대 모른다

**방법 B: 찾아보고 답하기 (RAG)**

RAG(Retrieval-Augmented Generation)는 마치 오픈북 시험처럼 작동합니다.

```
질문 들어옴
    ↓
관련 문서를 데이터베이스에서 검색 (Retrieval)
    ↓
검색된 문서를 AI에게 제공 + "이걸 바탕으로 답해줘"
    ↓
AI가 근거 기반으로 답변 생성 (Generation)
```

**비유:** 의사가 환자 차트(문서)를 보면서 진단하는 것 vs. 기억만으로 진단하는 것.

### 일반 RAG의 검색 방식: 코사인 유사도

문서와 질문을 숫자 벡터로 변환한 뒤, 방향이 비슷한(유사한) 문서를 찾습니다.

```
질문: "아인슈타인은 언제 태어났나?"
→ 벡터: [0.2, 0.8, 0.1, ...]

문서A: "아인슈타인은 1879년 출생..."  → 벡터 유사도: 0.92  ← 검색됨
문서B: "에펠탑은 1887년 착공..."      → 벡터 유사도: 0.21  ← 검색 안 됨
```

> 이 유사도 계산을 `sentence-transformers` 라이브러리가 합니다.  
> 이 프로젝트에서는 `all-MiniLM-L6-v2` 모델을 사용합니다.

## 2. GraphRAG — 그래프를 더하면 무엇이 달라지나?

### 일반 RAG의 한계: 멀티홉 질문

"아인슈타인이 받은 노벨상은 어느 나라에서 만들어진 상인가?"

이걸 답하려면:
1. 아인슈타인 → 노벨 물리학상 수상 (1921)
2. 노벨상 → 스웨덴 한림원이 수여

**두 번 이상 정보를 연결(hop)해야** 합니다. 일반 RAG는 각 문서를 독립적으로 보기 때문에 이런 연결을 만들기 어렵습니다.

### 지식 그래프(Knowledge Graph)란?

정보를 **노드(엔티티)**와 **엣지(관계)**로 표현한 네트워크입니다.

```
[Albert Einstein] --[received]--> [Nobel Prize in Physics]
[Nobel Prize in Physics] --[awarded_by]--> [Swedish Academy]
[Swedish Academy] --[located_in]--> [Sweden]
```

이렇게 그래프가 있으면 "아인슈타인 → 노벨상 → 스웨덴"을 경로로 찾을 수 있습니다.

### GraphRAG의 핵심 아이디어

RAG + 지식 그래프 = **검색한 문서에서 그래프를 만들고, 그 그래프를 따라 추론**

## 3. Multimodal GraphRAG — 이 논문이 푸는 문제

### 기존 GraphRAG의 추가 한계

기존 GraphRAG는 **텍스트만** 다룹니다. 하지만 실제 세계의 정보는:

- 📝 텍스트 (Wikipedia 문단)
- 📊 테이블 (노벨상 수상자 목록, 재무 데이터)
- 🖼️ 이미지 (건물 사진, 인물 사진)

이 세 가지가 뒤섞여 있습니다. "에펠탑의 높이는?"이라는 질문은 테이블에서 더 정확한 답을 얻을 수 있죠.

### 이 논문의 두 가지 핵심 기여

**기여 1 — 쿼리 주도(Query-Driven) 그래프 구축**

기존: 전체 문서를 미리 그래프화해서 저장 (시간·비용 매우 큼)  
이 논문: 질문이 들어올 때만, 그 질문에 필요한 그래프만 동적으로 구축

**기여 2 — 멀티모달 통합**

텍스트·테이블·이미지를 하나의 그래프에 통합해서 추론

## 5. Step 1 — Query Analysis (쿼리 분석)

**목표:** 질문을 분석해서 "어떤 종류의 정보를, 어느 모달리티에서, 몇 단계로 찾아야 하는가"를 담은 **Pattern Graph**를 만든다.

### 핵심 개념: Pattern Graph

Pattern Graph는 검색·그래프 구축의 "설계도"입니다.

```python
@dataclass
class PatternGraph:
    question_type: str           # 질문 유형: what / who / when / where / how / yes_no
    required_modalities: list    # 필요한 모달리티: ["text", "table", "image"]
    entity_types: list           # 기대하는 엔티티: ["Person", "Award", "Organization"]
    relation_types: list         # 기대하는 관계: ["received", "awarded_by"]
    hop_count: int               # 추론 단계 수: 1 (직접), 2~3 (멀티홉)
    focus_entity: str            # 핵심 엔티티: "Albert Einstein"
    subqueries: list             # 분해된 서브쿼리
    raw_query: str               # 원본 질문
```

### 코드 분석: `src/query_analyzer.py`

```python
class QueryAnalyzer:
    def analyze(self, query: str) -> PatternGraph:
        prompt = f"""You are an expert at analyzing questions...

Analyze the following question and extract a structured pattern graph...

Question: "{query}"

Return a JSON object with exactly these fields:
{{
  "question_type": "<what|who|when|where|how|yes_no|count>",
  "required_modalities": ["<text|image|table>"],
  "entity_types": [...],
  "relation_types": [...],
  "hop_count": <1-3>,
  "focus_entity": "...",
  "subqueries": [...]
}}"""

        response = self.client.messages.create(
            model=self.model,
            max_tokens=512,
            messages=[{"role": "user", "content": prompt}]
        )
        data = json.loads(response.content[0].text)
        return PatternGraph(**data)
```

**포인트 해설:**

| 코드 부분 | 설명 |
|---|---|
| `self.client.messages.create(...)` | Claude API를 호출해서 질문 분석을 맡김 |
| `max_tokens=512` | 응답이 JSON이므로 짧게 제한 (비용 절약) |
| `json.loads(...)` | Claude가 반환한 JSON 문자열을 Python 딕셔너리로 변환 |
| `re.sub(r"^```json\s*", "", raw)` | Claude가 가끔 ` ```json ` 코드블록으로 감싸므로 제거 |

### 실제 예시

```
입력 질문: "Who designed the Eiffel Tower and what was the occasion for which it was built?"

출력 Pattern Graph:
  question_type: "who"
  required_modalities: ["text"]
  entity_types: ["Person", "Building", "Event"]
  relation_types: ["designed", "built_for", "occurred_in"]
  hop_count: 2
  focus_entity: "Eiffel Tower"
  subqueries: [
    "Who designed the Eiffel Tower?",
    "What occasion was the Eiffel Tower built for?"
  ]
```

hop_count가 2인 이유: "설계자 찾기" + "건축 목적 찾기" 두 단계가 필요하기 때문입니다.

## 6. Step 2 — Multi-path Retrieval (다중 경로 검색)

**목표:** Pattern Graph를 바탕으로 텍스트·테이블·이미지 세 경로에서 동시에 관련 문서를 검색한다.

### 핵심 개념: 임베딩 유사도 검색

텍스트를 숫자 벡터로 변환하고, 내적(dot product)으로 유사도를 계산합니다.

```
"Eiffel Tower" 쿼리 벡터:        [0.3, 0.9, 0.1, 0.7, ...]
문서 "에펠탑은 파리에...":          [0.3, 0.8, 0.2, 0.6, ...]  → 유사도: 높음
문서 "아인슈타인은 물리학자...":    [0.8, 0.1, 0.9, 0.2, ...]  → 유사도: 낮음
```

### 코드 분석: `src/retriever.py`

**인덱싱 (사전 준비 단계)**

```python
def index_texts(self, docs: list[dict]) -> None:
    self._text_docs = docs
    texts = [d["content"] for d in docs]
    self._text_embeddings = self.embedder.encode(texts, convert_to_numpy=True)
    # 결과: shape=(문서수, 384) 짜리 numpy 행렬
    # 384는 all-MiniLM-L6-v2 모델의 벡터 차원수
```

`self.embedder.encode(texts)` 는 문장 리스트를 받아서 각각을 384차원 벡터로 변환합니다.  
이 행렬을 메모리에 저장해두면 나중에 빠르게 검색할 수 있습니다.

**텍스트 검색**

```python
def _retrieve_text(self, query: str, top_k: int) -> list[RetrievedDoc]:
    q_emb = self.embedder.encode([query], convert_to_numpy=True)  # 쿼리도 벡터화
    scores = (self._text_embeddings @ q_emb.T).flatten()          # 행렬 곱 = 모든 유사도 한번에 계산
    top_idx = np.argsort(scores)[::-1][:top_k]                    # 높은 순으로 정렬
    return [RetrievedDoc(doc_id=..., score=float(scores[i]), ...) for i in top_idx]
```

| 코드 | 의미 |
|---|---|
| `self._text_embeddings @ q_emb.T` | (문서수×384) × (384×1) = (문서수×1) 행렬. 한 번에 전체 유사도 계산 |
| `np.argsort(scores)[::-1]` | 점수를 낮은 순으로 정렬한 인덱스 → `[::-1]`로 뒤집어 높은 순으로 |

**멀티 경로 검색의 핵심**

```python
def retrieve(self, pattern: PatternGraph, top_k: int = 5) -> list[RetrievedDoc]:
    results = []

    # 서브쿼리마다 텍스트 검색 (멀티홉 대응)
    for subquery in pattern.subqueries:
        results.extend(self._retrieve_text(subquery, top_k))

    # 패턴이 이미지를 요구하면 이미지도 검색
    if "image" in pattern.required_modalities:
        results.extend(self._retrieve_images(pattern.raw_query, top_k))

    # 패턴이 테이블을 요구하면 테이블도 검색
    if "table" in pattern.required_modalities:
        results.extend(self._retrieve_tables(pattern.raw_query, top_k))

    # 중복 제거 후 점수 내림차순 반환
    seen = set()
    unique = []
    for doc in sorted(results, key=lambda x: x.score, reverse=True):
        if doc.doc_id not in seen:
            seen.add(doc.doc_id)
            unique.append(doc)
    return unique[:top_k * 2]
```

**왜 서브쿼리마다 검색할까?**

"Who designed the Eiffel Tower and what was the occasion?" 라는 질문을:
- 서브쿼리 1: "Who designed the Eiffel Tower?" → 귀스타브 에펠 관련 문서 검색
- 서브쿼리 2: "What occasion was Eiffel Tower built for?" → 1889 만국박람회 관련 문서 검색

원본 쿼리만으로 검색하면 어느 한쪽 정보만 찾힐 수 있습니다.

### RetrievedDoc 데이터 구조

```python
@dataclass
class RetrievedDoc:
    doc_id: str          # "t001", "tbl002" 등 문서 ID
    content: str         # 문서 내용 (텍스트 or 테이블 마크다운)
    modality: str        # "text" / "image" / "table"
    score: float         # 유사도 점수 (0~1)
    metadata: dict       # 출처 등 부가 정보
    image_path: str      # 이미지면 파일 경로
    image_b64: str       # Claude API에 넘길 base64 인코딩 이미지
```

## 7. Step 3 — Local KG Construction (지식 그래프 구축)

**목표:** 검색된 문서들에서 엔티티(노드)와 관계(엣지)를 추출해 이 질문 전용 지식 그래프를 만든다.

### 핵심 개념: 지식 트리플 (Triple)

지식 그래프는 **주어 - 서술어 - 목적어** 형태의 트리플로 구성됩니다.

```
(Albert Einstein) --[received]--> (Nobel Prize in Physics)
(Nobel Prize in Physics) --[awarded_year]--> (1921)
(Nobel Prize in Physics) --[reason]--> (Photoelectric effect)
```

### 코드 분석: `src/kg_builder.py`

**트리플 추출 (텍스트)**

```python
def _extract_triples_from_text(self, doc: RetrievedDoc, pattern: PatternGraph):
    prompt = f"""Extract knowledge graph triples from the text below.
Focus on entity types: {pattern.entity_types}         ← Step1에서 만든 설계도 활용
Focus on relation types: {pattern.relation_types}     ← Step1에서 만든 설계도 활용
Core entity of interest: "{pattern.focus_entity}"

Text: \"\"\"{doc.content[:1500]}\"\"\"

Return a JSON array of triples:
[{{"subject": "...", "subject_type": "...", "predicate": "...", "object": "...", "object_type": "..."}}]
"""
```

**포인트:** Pattern Graph(Step 1 결과)를 프롬프트에 넣어서, 관련 없는 트리플이 추출되지 않도록 유도합니다. "에펠탑" 관련 질문인데 "아인슈타인의 생일" 트리플을 추출하지 않도록요.

**트리플 → 그래프 추가**

```python
@staticmethod
def _add_triple(graph: nx.DiGraph, triple: dict, source_doc: str) -> None:
    subj = triple.get("subject", "").strip()   # 주어: "Albert Einstein"
    pred = triple.get("predicate", "").strip() # 서술어: "received"
    obj  = triple.get("object", "").strip()    # 목적어: "Nobel Prize in Physics"

    if subj not in graph:
        graph.add_node(subj, entity_type=triple.get("subject_type", "Unknown"))
    if obj not in graph:
        graph.add_node(obj, entity_type=triple.get("object_type", "Unknown"))

    graph.add_edge(subj, obj, relation=pred, source_doc=source_doc)
```

`nx.DiGraph`는 방향이 있는 그래프입니다. "Einstein → received → Nobel Prize"처럼 방향이 중요하기 때문에 유향 그래프(DiGraph)를 사용합니다.

**이미지에서 트리플 추출 (멀티모달)**

```python
def _extract_triples_from_image(self, doc: RetrievedDoc, pattern: PatternGraph):
    messages = [{
        "role": "user",
        "content": [
            {
                "type": "image",
                "source": {
                    "type": "base64",
                    "media_type": "image/jpeg",
                    "data": doc.image_b64,   # 이미지를 base64로 Claude에 전송
                },
            },
            {
                "type": "text",
                "text": f"Extract knowledge graph triples from this image..."
            },
        ],
    }]
    response = self.client.messages.create(model=self.model, messages=messages)
```

Claude의 멀티모달 기능 — 이미지와 텍스트를 함께 하나의 메시지로 전달해서 이미지에서도 트리플을 추출합니다.

### 전체 build 흐름

```python
def build(self, retrieved_docs: list[RetrievedDoc], pattern: PatternGraph) -> nx.DiGraph:
    graph = nx.DiGraph()

    for doc in retrieved_docs:
        if doc.modality == "text":
            triples = self._extract_triples_from_text(doc, pattern)
        elif doc.modality == "table":
            triples = self._extract_triples_from_table(doc, pattern)
        elif doc.modality == "image":
            triples = self._extract_triples_from_image(doc, pattern)

        for triple in triples:
            self._add_triple(graph, triple, doc.doc_id)

    return graph
```

문서 모달리티에 따라 다른 추출 방법을 사용하지만, 결국 모두 같은 그래프에 트리플로 쌓입니다. 텍스트와 테이블과 이미지 정보가 하나의 그래프 안에 통합되는 것이 핵심입니다.

## 8. Step 4 — Multimodal Supplement (멀티모달 정보 보충)

**목표:** Step 3에서 만든 그래프에서 이미지·테이블 정보가 부족한 노드를 찾아 추가로 보충한다.

### 왜 필요한가?

Step 3의 검색에서 텍스트 문서로 노드가 만들어졌는데, 그 노드가 "Person"이나 "Building"이라면 시각적 정보(이미지)나 수치 정보(테이블)가 있으면 더 좋은 답변이 가능합니다.

예: "Albert Einstein" 노드 → 사진에서 "흰머리 노인, 물리학자 풍모" 같은 시각 정보 추가

### 코드 분석: `src/multimodal_supplement.py`

```python
def supplement(self, graph: nx.DiGraph, pattern: PatternGraph) -> nx.DiGraph:
    # 이미지·테이블이 필요 없는 질문이면 그냥 통과
    if "image" not in pattern.required_modalities and "table" not in pattern.required_modalities:
        return graph

    # 시각 정보가 도움될 만한 엔티티 타입의 노드만 선별
    nodes_needing_visual = [
        n for n, data in graph.nodes(data=True)
        if data.get("modality") == "text"
        and data.get("entity_type") in ("Person", "Product", "Building", "Animal", "Artwork")
    ]

    for node_name in nodes_needing_visual[:3]:   # API 비용 절약을 위해 최대 3개
        if "image" in pattern.required_modalities:
            self._add_image_info(graph, node_name)
        if "table" in pattern.required_modalities:
            self._add_table_info(graph, node_name)

    return graph
```

**테이블 정보 보충**

```python
def _add_table_info(self, graph: nx.DiGraph, entity: str) -> None:
    table_docs = self.retriever._retrieve_tables(entity, top_k=1)  # 이 엔티티와 관련된 테이블 검색
    if not table_docs:
        return
    graph.nodes[entity]["table_data"] = table_docs[0].content[:500]  # 노드 속성에 저장
    graph.nodes[entity]["modality"] = "multimodal"                   # 모달리티 업데이트
```

**이미지 보충 (Claude 비전 활용)**

```python
def _describe_image(self, image_b64: str, entity: str) -> str:
    response = self.client.messages.create(
        model=self.model,
        max_tokens=200,
        messages=[{
            "role": "user",
            "content": [
                {"type": "image", "source": {"type": "base64", "data": image_b64}},
                {"type": "text", "text": f"Describe what you see about '{entity}' in 1-2 sentences."},
            ],
        }]
    )
    return response.content[0].text
```

이미지를 Claude에게 보내서 자연어 설명을 생성하고, 그 설명을 노드 속성 `visual_description`에 저장합니다. 이후 Step 5에서 이 설명이 추론에 활용됩니다.

## 9. Step 5 — Graph-based Reasoning (그래프 추론)

**목표:** 완성된 지식 그래프를 경로 탐색하고, 그 경로를 Claude에게 제공해 최종 답변을 생성한다.

### 핵심 개념: 멀티홉 경로 탐색

```
질문: "아인슈타인이 받은 상을 수여하는 기관은?"

그래프 경로:
[Albert Einstein] → received → [Nobel Prize in Physics]
                                        ↓ awarded_by
                               [Swedish Academy of Sciences]

최단 경로: Einstein → Nobel Prize → Swedish Academy (2-hop)
```

### 코드 분석: `src/reasoner.py`

**그래프 직렬화 (텍스트로 변환)**

```python
def _serialize_graph(self, graph: nx.DiGraph, pattern: PatternGraph) -> str:
    lines = ["[Knowledge Graph Triples]"]

    focus = pattern.focus_entity.lower()
    relevant_nodes = set()

    # 핵심 엔티티 주변 노드만 먼저 수집
    for node in graph.nodes():
        if focus and focus in node.lower():
            relevant_nodes.add(node)
            relevant_nodes.update(graph.successors(node))    # 아래 이웃
            relevant_nodes.update(graph.predecessors(node))  # 위 이웃

    count = 0
    for u, v, data in graph.edges(data=True):
        if u in relevant_nodes or v in relevant_nodes:
            rel = data.get("relation", "related_to")
            lines.append(f"  ({u}) --[{rel}]--> ({v})")
            count += 1
            if count >= 40:  # 토큰 절약
                break

    return "\n".join(lines)
```

그래프를 직접 LLM에 넣을 수 없으므로, 트리플 형태의 텍스트로 직렬화해서 프롬프트에 포함시킵니다.

**멀티홉 경로 탐색**

```python
def _find_relevant_paths(self, graph: nx.DiGraph, pattern: PatternGraph) -> list:
    paths = []
    focus = pattern.focus_entity.lower()

    # 핵심 엔티티로 시작하는 노드들 찾기
    start_nodes = [n for n in graph.nodes() if focus and focus in n.lower()]

    for start in start_nodes[:2]:
        for end in graph.nodes():
            if start == end:
                continue
            try:
                path = nx.shortest_path(graph, start, end)
                # hop_count 이하의 경로만 포함
                if 1 < len(path) <= pattern.hop_count + 1:
                    paths.append(path)
            except nx.NetworkXNoPath:
                pass

    return paths[:10]
```

`nx.shortest_path(graph, start, end)` — NetworkX 라이브러리의 최단 경로 탐색 함수입니다. BFS(너비 우선 탐색) 알고리즘으로 그래프 경로를 찾습니다.

**최종 답변 생성**

```python
def _generate_answer(self, query, graph_text, paths, pattern) -> dict:
    path_text = "\n[Reasoning Paths]\n"
    for path in paths[:5]:
        path_text += "  " + " → ".join(path) + "\n"
    # 예시: "Albert Einstein → Nobel Prize in Physics → 1921"

    prompt = f"""Use the knowledge graph below to answer the question.

Question: "{query}"

{graph_text}     ← 그래프 트리플 텍스트
{path_text}      ← 발견된 경로들

Return JSON:
{{
  "answer": "...",
  "reasoning_chain": ["step1", "step2", ...],
  "evidence_nodes": ["node1", "node2"],
  "confidence": 0.0~1.0
}}"""
```

**ReasoningResult 구조**

```python
@dataclass
class ReasoningResult:
    answer: str                  # 최종 답변 텍스트
    reasoning_chain: list[str]   # 추론 과정 단계별 설명
    evidence_nodes: list[str]    # 답변 근거가 된 노드들
    evidence_edges: list[tuple]  # 사용된 엣지들
    confidence: float            # 신뢰도 (0.0 ~ 1.0)
```

## 11. 일반 RAG vs GraphRAG 비교

| 항목 | 일반 RAG | 이 프로젝트 (Multimodal GraphRAG) |
|---|---|---|
| **검색 방식** | 질문 전체로 단일 검색 | 서브쿼리로 분해 후 다중 경로 검색 |
| **모달리티** | 텍스트만 | 텍스트 + 테이블 + 이미지 통합 |
| **문서 관계** | 각 문서를 독립적으로 봄 | 문서 간 관계를 그래프로 연결 |
| **멀티홉 추론** | 어려움 | 그래프 경로 탐색으로 자연스럽게 해결 |
| **근거 추적** | 어느 문서를 썼는지만 | 어떤 경로를 거쳐 답했는지 명시 |
| **그래프 구축 시점** | 없음 | 질문이 들어올 때 동적으로 구축 |
| **API 호출 횟수** | 1~2회 | 단계별 여러 번 (더 정확한 대신 비용 높음) |

### 멀티홉 질문에서의 차이

질문: "Steve Jobs가 투자한 애니메이션 스튜디오의 가장 유명한 작품은?"

**일반 RAG 흐름:**
```
검색: "Steve Jobs animation studio famous movie"
→ 운이 좋으면 Pixar·Toy Story 문서가 함께 검색됨
→ 운이 나쁘면 Apple 관련 문서만 검색됨
```

**GraphRAG 흐름:**
```
서브쿼리 1: "Steve Jobs animation studio" → Steve Jobs → invested_in → Pixar
서브쿼리 2: "Pixar famous movies" → Pixar → produced → Toy Story
경로: Steve Jobs → Pixar → Toy Story
답변: "Toy Story (1995)"
```

## 12. 핵심 용어 정리

| 용어 | 설명 |
|---|---|
| **RAG** | Retrieval-Augmented Generation. 문서를 검색해서 LLM에 제공하는 방식 |
| **LLM** | Large Language Model. GPT, Claude 같은 대형 언어 모델 |
| **임베딩 (Embedding)** | 텍스트를 숫자 벡터로 변환하는 것. 의미가 비슷한 문장은 벡터도 비슷함 |
| **코사인 유사도** | 두 벡터가 얼마나 같은 방향을 가리키는지. 1에 가까울수록 유사 |
| **지식 그래프 (KG)** | 엔티티(노드)와 관계(엣지)로 정보를 표현한 그래프 |
| **트리플** | (주어, 서술어, 목적어) 형태의 지식 단위. 그래프의 기본 요소 |
| **멀티홉 추론** | 여러 단계의 관계를 거쳐 답에 도달하는 추론 방식 |
| **Pattern Graph** | 이 논문의 핵심 개념. 질문에서 추출한 검색·구축 설계도 |
| **모달리티 (Modality)** | 정보의 형태. 텍스트·이미지·테이블 각각이 하나의 모달리티 |
| **DiGraph** | 방향 있는 그래프 (Directed Graph). NetworkX에서 `nx.DiGraph()` |
| **BFS** | Breadth-First Search. 최단 경로를 찾는 너비 우선 탐색 알고리즘 |
| **Sentence Transformers** | 문장을 벡터로 변환하는 오픈소스 라이브러리 |
| **SSE** | Server-Sent Events. 서버에서 클라이언트로 실시간 데이터를 스트리밍하는 기술 |
| **할루시네이션** | LLM이 사실이 아닌 내용을 자신있게 답하는 현상 |
| **신뢰도 (Confidence)** | LLM이 자체 평가한 답변의 확신도 (0.0~1.0). 절대적 정확도와는 다름 |
